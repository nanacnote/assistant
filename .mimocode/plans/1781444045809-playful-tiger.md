# Assistant Roadmap: World-Class Second Brain

## Current State Assessment

**What exists (Alpha, v0.1.0):**
- Matrix-based AI assistant with event-driven architecture (Beacon + Keel)
- Agentic tool-calling loop with self-correction
- Memory: working memory, long-term fact extraction, procedure memory (PostgreSQL + full-text search)
- Builtin tools: Reply, Calendar (CRUD + recurrence + reminders), Wellbeing coaching, AskUser (interactive clarification), Procedure memory (save/plan)
- Plugin architecture for custom tool modules
- Docker Compose deployment (Synapse + Postgres + Nginx + Element Web)
- Grounding context (temporal + identity awareness)

**Current gaps:**
- Single shared DB connection (no connection pooling)
- No streaming LLM responses
- No multi-modal support (voice, images, files)
- No proactive capabilities (scheduled tasks, reminders don't fire automatically)
- Memory search is PostgreSQL full-text only (no semantic/vector search)
- No web dashboard or REST API
- No authentication/authorization beyond Matrix identity
- No observability (metrics, tracing, structured logging)
- Limited error recovery (no retry logic, no circuit breakers)
- No multi-LLM support (single provider)
- TODO: Pending question TTL cleanup not wired

## Phase 1: Foundation Hardening (Weeks 1-4)

### 1.1 Database Connection Pooling
- **Why:** Current single shared connection is a bottleneck and fragile
- **How:** Replace `get_shared_connection()` with `psycopg` connection pool (`ConnectionPool`)
- **Files:** `assistant/storage/connection.py`, all repository files
- **Impact:** Reliability, concurrent request handling

### 1.2 Retry Logic & Circuit Breakers
- **Why:** LLM API and Matrix API calls can transiently fail
- **How:** Add exponential backoff retries to `JsonHttpClient`, circuit breaker for LLM calls
- **Files:** `assistant/http_client.py`, `assistant/llm.py`
- **Impact:** Reliability under partial outages

### 1.3 Streaming LLM Responses
- **Why:** Users wait for full response; streaming feels instant
- **How:** Support SSE streaming from OpenAI-compatible endpoints, stream tokens to Matrix via message edits
- **Files:** `assistant/llm.py`, `assistant/matrix.py`, `assistant/runtime.py`
- **Impact:** UX, perceived responsiveness

### 1.4 Structured Logging & Observability
- **Why:** Current logging is basic; no request tracing
- **How:** Add structlog, request ID propagation, tool execution timing metrics
- **Files:** `assistant/main.py`, `assistant/runtime.py`, all tool modules
- **Impact:** Debuggability, production readiness

### 1.5 Wire Pending Question TTL Cleanup
- **Why:** Stale questions accumulate in DB (existing TODO)
- **How:** Use a periodic async task or Beacon scheduled event to call `delete_expired()`
- **Files:** `assistant/runtime.py`, `assistant/interaction/repository.py`
- **Impact:** Data hygiene

### 1.6 Comprehensive Test Coverage
- **Why:** Tests exist but coverage is partial
- **How:** Add integration tests for the full agentic loop, memory extraction, tool execution
- **Files:** `tests/`, `assistant/tests/`
- **Impact:** Confidence in changes, regression prevention

## Phase 2: Memory System Evolution (Weeks 5-8)

### 2.1 Semantic Memory Search (Vector Embeddings)
- **Why:** PostgreSQL full-text search misses semantic similarity
- **How:** Add embedding generation (OpenAI embeddings or local model), pgvector extension for similarity search, hybrid search (BM25 + vector)
- **Files:** `assistant/memory/repository.py`, `assistant/memory/service.py`, `storage/migrations.py`
- **Impact:** Dramatically better memory recall

### 2.2 Memory Consolidation & Summarization
- **Why:** Raw facts accumulate; need periodic distillation
- **How:** Scheduled job that clusters related facts and creates summary memories, merges duplicates
- **Files:** `assistant/memory/service.py`, new `assistant/memory/consolidation.py`
- **Impact:** Memory quality, reduced noise

### 2.3 Episodic Memory (Conversation Summaries)
- **Why:** Working memory prunes old messages; summaries preserve context
- **How:** After N turns, generate a conversation summary and store as a special memory type
- **Files:** `assistant/memory/service.py`, `assistant/memory/models.py`
- **Impact:** Long-term conversation continuity

### 2.4 User Knowledge Graph
- **Why:** Facts about people, places, projects have relationships
- **How:** Store entity relationships alongside facts, enable graph-style queries ("who works with Alice on Project X?")
- **Files:** `assistant/memory/models.py`, `assistant/memory/repository.py`, `storage/migrations.py`
- **Impact:** Richer contextual understanding

### 2.5 Memory Confidence & Source Tracking
- **Why:** Some memories are more reliable than others
- **How:** Add confidence scores, source attribution (user stated vs. inferred), contradiction detection
- **Files:** `assistant/memory/models.py`, `assistant/memory/extraction.py`
- **Impact:** Trustworthiness of recalled information

## Phase 3: Proactive Intelligence (Weeks 9-12)

### 3.1 Scheduled Task Engine

- **Impact:** Proactive reminders, daily briefings, automated check-ins

### 3.2 Proactive Wellbeing Check-ins

### 3.3 Calendar Intelligence

- **Files:** `assistant/builtin_tools/calendar/`, `assistant/scheduler.py`
- **Impact:** Time management assistant

### 3.4 Daily Briefing Generation
- **Why:** Users want a morning summary
- **How:** Scheduled task that compiles calendar, wellbeing trends, pending items into a daily digest
- **Files:** New `assistant/builtin_tools/briefing/`
- **Impact:** Executive assistant functionality

### 3.5 Context-Aware Follow-ups
- **Why:** Conversations end without resolution
- **How:** Track incomplete tasks, send follow-up messages after configurable delays
- **Files:** `assistant/interaction/`, `assistant/scheduler.py`
- **Impact:** Task completion, reliability

## Key Architecture Decisions Needed

1. **Vector DB choice:** pgvector (keep PostgreSQL) vs. dedicated (Qdrant, Weaviate, Pinecone)
2. **Streaming approach:** Matrix message edits vs. new messages per chunk
3. **Scheduler:** APScheduler vs. custom asyncio loop vs. Beacon scheduled events
4. **Web framework:** FastAPI vs. Starlette (lighter)
5. **Frontend:** React SPA vs. htmx server-rendered
6. **Multi-transport:** Abstract at Beacon level vs. parallel transport layer
