# Procedure Memory System ‚ÄĒ Implementation Plan

## Overview

Add a procedure memory system that stores "how to do X" task decompositions and makes them available to the LLM for future requests. Procedures are stored per-user, retrieved by relevance, and can be created both automatically (after successful multi-step tool chains) and explicitly (via `SaveProcedure` tool).

## Data Model

**New table** `procedure_memories` in `storage/migrations.py`:

```sql
CREATE TABLE IF NOT EXISTS procedure_memories (
    id              TEXT PRIMARY KEY,
    user_id         TEXT NOT NULL,
    description     TEXT NOT NULL,
    steps           JSONB NOT NULL DEFAULT '[]'::jsonb,
    category        TEXT NOT NULL DEFAULT 'general',
    importance      REAL NOT NULL DEFAULT 0.5,
    source_conv_id  TEXT NOT NULL DEFAULT '',
    execution_count INTEGER NOT NULL DEFAULT 0,
    success_count   INTEGER NOT NULL DEFAULT 0,
    access_count    INTEGER NOT NULL DEFAULT 0,
    last_accessed   TIMESTAMPTZ NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_procedure_memories_user
    ON procedure_memories (user_id, importance DESC, last_accessed DESC);
CREATE INDEX IF NOT EXISTS idx_procedure_memories_fts
    ON procedure_memories USING GIN (to_tsvector('english', description));
```

Steps stored as JSON array of natural-language strings:
```json
["Search for events matching the user's query", "Extract the event_id from results", "Call DeleteEvent with the event_id"]
```

## AskUser Tool ‚ÄĒ Clarifying Questions

### Overview

Add an `AskUser` tool that the LLM can call when it needs clarification before proceeding. The tool sends a question to the Matrix room, waits for the user's reply, and returns it to the LLM so it can continue the agentic loop with the new information.

**Example flow:**
1. User: "I want directions"
2. LLM calls `AskUser(question="Where would you like directions to?", room_id="!room:example.com")`
3. Tool sends the question to the Matrix room
4. Tool blocks, waiting for user reply (with timeout)
5. User: "To the airport"
6. Tool returns `{"answer": "To the airport"}`
7. LLM continues with the information and calls the appropriate tool

### How it works

The agentic loop runs in a background thread (`asyncio.to_thread`). Matrix sync runs in the asyncio event loop. The `AskUser` tool bridges these via `threading.Event`:

1. `AskUser.execute()` registers a pending question (keyed by `room_id`) with a `threading.Event` + reply slot
2. It sends the question to the Matrix room via `_run_async(matrix_client.send_text(...))`
3. It blocks on `threading.Event.wait(timeout=60)`
4. In `_handle_message_event`, before normal processing, check if the incoming message's `room_id` matches a pending question. If yes: store the reply, set the event, return (don't process as a new request)
5. The tool wakes up, returns the reply text to the LLM

**Concurrent requests:** Each agentic loop runs in its own thread. A waiting `AskUser` on one thread does not block other requests on other threads.

**Timeout:** If no reply within 60 seconds, returns `{"answer": null, "timeout": true}`. The LLM can then decide to proceed without the info or give up.

### Pydantic Strictness

Both `question` and `room_id` are required fields with no defaults. When the LLM omits them:
1. Pydantic raises `ValidationError` with `type: "missing"`
2. Keel catches `SchemaDispatchError` and generates a correction prompt
3. The correction prompt tells the LLM exactly which field is missing
4. LLM retries (up to 3 times)

The system prompt already marks required fields as `REQUIRED` and tells the LLM to include them. Good `Field(description=...)` text guides the LLM on what each field should contain.
