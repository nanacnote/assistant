# Assistant: Log Analysis & Improvement Plan

## Summary

The system is **not working as expected**. 5 critical issues, 2 minor. The LLM defaults to `Reply` every turn — never uses action tools (LogState, RecordCheckIn, GetInsights), producing a passive, repetitive conversation.

Keel is an **external package** — we cannot modify it here. This plan covers everything actionable in the assistant codebase. A companion plan (`keel-llm-steering.md`) covers what keel needs to change — it should be passed to the keel team.

**Dependency**: Issues 2–4 (LLM steering) require keel changes to fully resolve. The assistant-side prep work (tool metadata) can be done now and will activate when keel is updated.

---

## Issues & Solutions

### 1. CRITICAL: Stale "Tool results so far" pollutes every subsequent turn

**What happens**: After AskUser completes, the full reconstructed prompt (including tool history) is stored as the "user message" in working memory. On every subsequent turn, this stale block reappears:

```
user: 
Tool results so far:
  [step 1] AskUser → asked: What would you like to focus on...
User answered: my mental health
Continue your task using the user's answer. Respond with the Reply tool.
```

**Root cause** (`runtime.py:271-306`, `runtime.py:365-392`): `_reconstruct_prompt_with_answer()` builds the full prompt, then `_dispatch_via_keel` stores `current_user_msg` (the raw prompt string) and passes it to `store_turn()`. The reconstructed prompt artifacts end up in working memory.

**Fix** (`assistant/runtime.py`): In `_dispatch_via_keel`, detect AskUser resume requests and extract only the user's actual answer before storing:

```python
# After line 368 where current_user_msg is extracted:
if request.id.startswith("resume-"):
    for line in current_user_msg.split("\n"):
        if line.startswith("User answered:"):
            current_user_msg = line[len("User answered:"):].strip()
            break
```

**Blocked on keel?** No. Fully actionable now.

### 2–4. CRITICAL: LLM doesn't use action tools, asks repetitive questions, describes instead of acting

These three issues share the same root cause: the system prompt doesn't guide the LLM toward proactive tool usage.

**What we can do now** — add metadata annotations to tools so they're ready when keel implements the grouping/rendering changes:

**`assistant/builtin_tools/reply.py`:**
```python
class Reply(BaseTool):
    """Return the final assistant reply."""
    tool_role = "meta"   # keel will use this to group tools in the prompt
    tool_name = "Reply"
    ...
```

**`assistant/builtin_tools/interaction/tools.py`:**
```python
class AskUser(BaseTool):
    """Ask the user a clarifying question before proceeding."""
    tool_role = "meta"
    tool_name = "AskUser"
    ...
```

**`assistant/builtin_tools/wellbeing/tools.py`:**
```python
class LogState(BaseTool):
    """Log a quick mental-emotional state snapshot."""
    usage_hint = "Use when the user provides numerical ratings or describes their emotional state"
    ...

class RecordCheckIn(BaseTool):
    """Record a short structured check-in for wake, mid, or sleep sessions."""
    usage_hint = "Use when starting or ending a structured wellness session"
    ...

class GetInsights(BaseTool):
    """Summarize wellbeing trends from recent entries."""
    usage_hint = "Use before giving advice to ground recommendations in actual data"
    ...
```

**Note**: `tool_role` and `usage_hint` are ClassVars that keel's `BaseTool` doesn't declare yet. Setting them now is safe — Python treats them as regular class attributes, and keel ignores unknown attributes. When keel adds support (see `keel-llm-steering.md`), these will activate automatically with no further assistant changes.

**Blocked on keel?** The metadata annotations: no (safe to add now). The actual LLM behavior change: yes (requires keel's system prompt rewrite).

### 5. MODERATE: Fact extraction fails silently

**What happens**: At 14:35:42, fact extraction returns non-JSON, causing: `WARNING assistant.memory.extraction: failed to parse fact extraction response as JSON`. The first user response ("my mental health") has no facts extracted.

**Root cause** (`extraction.py:85-120`): `_parse_extraction_response` tries strict JSON parsing. If the LLM wraps JSON in prose (not fenced with triple backticks), it fails.

**Fix** (`assistant/memory/extraction.py`): Add a fallback that attempts to find a JSON array/object in the response text:

```python
# After the markdown fence extraction and before the json.loads try:
if not text.startswith(("[", "{")):
    for start_char, end_char in [("[", "]"), ("{", "}")]:
        start = text.find(start_char)
        end = text.rfind(end_char)
        if start >= 0 and end > start:
            text = text[start:end + 1]
            break
```

**Blocked on keel?** No.

### 6. MODERATE: History grows unboundedly within a session

**What happens**: Working memory fetches the last 20 messages. By the end of this 3-minute interaction, there are 16 messages. Each new prompt includes all prior messages, causing linear token growth.

**Fix** (`assistant/reactions.py`): Cap the history injected into prompts:

```python
if ctx.history:
    recent_history = ctx.history[-10:]  # cap at 10 messages
    messages.extend(recent_history)
```

**Blocked on keel?** No.

### 7. MINOR: Fact extraction stores redundant facts

**What happens**: Multiple similar facts stored:
- "The user's current mood is 8" (14:37:12)
- "The user's current mood rating is 8" (14:39:12)

**Fix** (`assistant/memory/repository.py` or `service.py`): Before storing a new fact, check for existing facts with similar text for the same user.

**Blocked on keel?** No.
