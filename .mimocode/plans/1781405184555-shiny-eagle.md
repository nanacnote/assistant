# Non-blocking AskUser: Persist & Resume Architecture

## Problem

`AskUser.execute()` blocks the agentic loop thread for up to 60 seconds via `threading.Event.wait()`. This holds an OS thread hostage, the full agentic loop context in memory, and prevents that worker from processing other requests. If the user takes longer than 60s, the question times out and the context is lost.

## Solution

Replace the blocking `threading.Event` pattern with a **persist-and-reevaluate** approach:

1. AskUser sends the question and returns **immediately** with a sentinel
2. The agentic loop detects the sentinel and exits early
3. The runtime persists the request context to PostgreSQL
4. The thread is freed — no resources held
5. When the user replies, the runtime loads context, reconstructs a prompt with the Q&A injected, and re-dispatches through the full agentic loop

### New Flow

```
User msg → _handle_message_event → _dispatch_via_keel → _agentic_loop_sync
                                                            ↓
                                                   AskUser.execute()
                                                            ↓
                                                   sends question, returns sentinel
                                                            ↓
                                                   loop detects sentinel, returns _AskUserWaiting
                                                            ↓
                                                   _dispatch_via_keel detects waiting
                                                            ↓
                                                   persists to pending_questions table
                                                            ↓
                                                   returns (no LLM response sent)
                                                            ↓
                                                   THREAD FREED

--- Later, user replies ---

User reply → _handle_message_event
                ↓
          checks pending_questions table (room_id + thread_root)
                ↓
          loads saved context
                ↓
          reconstructs prompt with Q&A + tool history
                ↓
          _dispatch_via_keel(new_prompt) → normal agentic loop → Reply
                ↓
          deletes row from pending_questions table
```

### 1. `assistant/storage/migrations.py` — Add `pending_questions` table

```sql
CREATE TABLE IF NOT EXISTS pending_questions (
    id              TEXT PRIMARY KEY,
    room_id         TEXT NOT NULL,
    thread_root     TEXT NOT NULL DEFAULT '',
    question        TEXT NOT NULL,
    original_prompt TEXT NOT NULL,
    tool_history    JSONB NOT NULL DEFAULT '[]'::jsonb,
    request_metadata JSONB NOT NULL DEFAULT '{}'::jsonb,
    created_at      TIMESTAMPTZ NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_pending_questions_room_thread
    ON pending_questions (room_id, thread_root);
```

### 4. `assistant/runtime.py` — Detect sentinel, persist, and handle reply

**Changes to `_agentic_loop_sync`**:
- After each tool execution, check if `last_result` is a dict with `__ask_user_waiting == True`
- If so, return a `_AskUserWaiting` dataclass (not a string) containing `question_id`, `original_prompt`, `tool_history`

**Changes to `_dispatch_via_keel`**:
- Check if `_agentic_loop_sync` returned a `_AskUserWaiting` instance
- If so: save to DB via `pending_question_repo.save()`, return without sending an LLM response
- Cleanup `_request_metadata[request.id]` as usual

**Changes to `_handle_message_event`**:
- Remove the in-memory `pending_questions` dict check
- Add DB lookup: `pending_question_repo.find_by_room_thread(room_id, thread_root)`
- If found: reconstruct prompt, delete DB row, re-dispatch through normal flow

**Changes to `__init__`**:
- Remove `self._pending_questions` dict
- Instantiate `PostgresPendingQuestionRepository`
- Pass it to `load_tool_modules` instead of `pending_questions`

**New method `_reconstruct_prompt_with_answer`**:
```python
def _reconstruct_prompt_with_answer(
    self,
    original_prompt: str,
    tool_history: list[tuple[int, str, str]],
    question: str,
    answer: str,
) -> str:
    # Build: original prompt + tool history showing AskUser + answer injection
    lines = [original_prompt, "", "Tool results so far:"]
    for step_num, name, result in tool_history:
        lines.append(f"  [step {step_num}] {name} → {result}")
    lines += [
        f"  [step {len(tool_history) + 1}] AskUser → asked: {question}",
        "",
        f"User answered: {answer}",
        "",
        "Continue your task using the user's answer. Respond with the Reply tool.",
    ]
    return "\n".join(lines)
```

## Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Storage | PostgreSQL (existing) | Already a dependency, existing migration pattern, no new infra |
| Resume strategy | Re-run from scratch | Simpler, no state serialization, acceptable token cost |
| Cleanup | TTL + manual (scheduler TODO) | Matches user's plan to add Beacon scheduled events later |
| Concurrency key | `(room_id, thread_root)` | Matches existing Matrix thread model |
| Prompt reconstruction | Full prompt + tool history + Q&A | Gives LLM enough context to continue naturally |
