# Plan: Inject Grounding Context into LLM Prompts

## Problem

LLMs have no temporal or environmental awareness. The assistant sends prompts with no date, time, day-of-week, or system metadata -- so the LLM can't reason about "tomorrow", "next Monday", "today", or know what model it is.

## Approach

Create a single `build_grounding_context()` helper and inject it into both LLM prompt pipelines (main requests + memory extraction).

## Changes

### 1. New file: `assistant/grounding.py`

Pure function that builds a grounding block:

```python
def build_grounding_context(*, model: str = "", actor_id: str = "") -> str:
    now = datetime.now(timezone.utc)
    # Returns:
    # Current date and time: 2026-06-13T14:30:00Z (Saturday)
    # User: @alice:matrix.org
    # Model: gpt-4o
```

- UTC ISO 8601 + day-of-week (critical for relative time references)
- Optional model name and actor identity

### 2. `assistant/runtime.py` -- `_render_request()` (line ~401)

Inject grounding block after `conversation_id`/`actor_id` header, before memory system messages. This is the single per-request funnel before keel -- grounding is always fresh.

### 3. `assistant/memory/extraction.py`

Add `grounding: str = ""` parameter to both `extract_facts()` and `extract_procedure()`. Prepend to system prompt when provided. This makes extracted facts time-aware ("User has a meeting on 2026-06-14" instead of "tomorrow").

### 4. `assistant/memory/service.py`

Generate grounding in `extract_and_store_facts()` and `extract_and_store_procedure()` via `build_grounding_context(actor_id=user_id)` and pass to extraction functions.

## What the LLM sees

```
conversation_id: !abc:matrix.org
actor_id: @alice:matrix.org

Current date and time: 2026-06-13T14:30:00Z (Saturday)
User: @alice:matrix.org
Model: gpt-4o

Relevant memories about this user:
...
```

## Other grounding context to consider

| Context | Value | Notes |
|---------|-------|-------|
| Current date/time (UTC) | Critical | Enables "tomorrow", "next week", time-aware scheduling |
| Day of week | Critical | Included in the date line |
| Model name | Helpful | Lets LLM know its own identity/capabilities |
| Actor ID | Already available | Matrix user ID for identity |

Beyond these, the existing system prompt (via keel) already provides the tool catalog. User timezone could be useful but isn't consistently available (only calendar domain stores it). Best left for a future enhancement where the user's timezone preference is stored in memory.
