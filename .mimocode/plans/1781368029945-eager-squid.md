# Fix: Assistant not responding after memory module addition

## Root cause

Two issues combine to break the assistant:

### Issue 1: Startup crash (fatal)
`AssistantRuntime.__init__` at `runtime.py:75` calls `build_memory_service()` which calls `build_memory_repository()` (`ports.py:73-83`). That function calls `get_shared_connection()` → `psycopg.connect()` and `run_migrations()` — **all with zero error handling**. If the `assistant_tools` database doesn't exist (the postgres init script only runs on first volume creation) or postgres isn't fully ready, the entire runtime crashes before it ever starts listening for Matrix events. Docker restarts it, it crashes again, repeat.

### Issue 2: Event loop blocking (causes silent hangs)
Even when the DB is available, all `PostgresMemoryRepository` methods use **synchronous** psycopg calls. The `history_fetcher` callback (called by `ContextBuilder.build()` at `context_builder.py:62`) invokes `get_working_memory()` → `get_history()` which is a blocking DB query **not wrapped in `asyncio.to_thread()`**. Same for `get_relevant_memories()` → `search_facts()` / `touch_fact()`. If the DB is slow or the connection pool is exhausted, these block the entire async event loop, preventing any messages from being processed.

## Fix plan (5 changes)

### 1. Wrap `build_memory_service()` in try/except — `assistant/runtime.py:75-80`

Fallback to `UnconfiguredMemoryRepository` if DB is unreachable. The assistant must always start and respond to messages even without memory.

```python
try:
    self.memory_service = build_memory_service(...)
except Exception as exc:
    logger.warning("Memory service unavailable, continuing without memory: %s", exc)
    from assistant.memory.service import MemoryService
    from assistant.memory.ports import UnconfiguredMemoryRepository
    self.memory_service = MemoryService(
        repository=UnconfiguredMemoryRepository(),
        llm_complete=self.llm.complete,
        working_memory_limit=settings.memory.working_memory_limit,
        max_facts_per_user=settings.memory.max_facts_per_user,
        extraction_enabled=False,
    )
```

### 2. Add retry logic — `assistant/memory/ports.py:73-83`

3 attempts with 2s backoff in `build_memory_repository()` to handle postgres startup race.

### 3. Add `connect_timeout` — `assistant/storage/connection.py:24`

Change `connect(resolved_dsn, row_factory=dict_row)` to `connect(resolved_dsn, row_factory=dict_row, connect_timeout=10)`.

### 4. Wrap sync DB calls in `asyncio.to_thread()` — `assistant/memory/service.py`

All async methods that call synchronous repo methods need wrapping:
- `get_working_memory()` line 39: wrap `self._repo.get_history()` 
- `get_relevant_memories()` line 79: wrap `self._repo.search_facts()` and `self._repo.touch_fact()`
- `store_turn()` lines 51-71: wrap `self._repo.store_message()` and `self._repo.prune_conversation()`
- `extract_and_store_facts()` lines 106-124: wrap `self._repo.store_fact()` and `self._repo.prune_old_facts()`

### 5. Add test for graceful fallback — `tests/test_runtime.py`

Test that `AssistantRuntime` starts and responds to messages when the memory module fails.

## Files to modify

| File | Change |
|---|---|

## Verification

1. Run `pytest tests/` — all existing tests pass
2. With a bad DSN: assistant starts and responds to messages (without memory)
3. With a valid DSN + `assistant_tools` DB: memory works end-to-end
4. `docker compose logs assistant` shows no crash loops
