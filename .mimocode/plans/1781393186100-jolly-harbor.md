# Grounding Context: Assistant-side, not Keel-side

## Recommendation

**Implement in the assistant.** Keel is the wrong layer for this.

## Why not Keel

Keel (`keel/core/`) is a thin, single-turn tool-calling engine. It has:

- **No concept of temporal context, user identity, or model awareness.** Its `prompts.py` generates only a tool catalog + format instructions. Zero references to datetime, timezone, or grounding anywhere in its source.
- **A single injection seam**: `Engine.run(user_message, system_prompt)` — the caller provides the system prompt. Keel doesn't compose it.
- **No middleware around LLM calls.** Its `MiddlewareChain` wraps `tool.execute()`, not prompt assembly.

Adding grounding to keel would couple a generic, reusable tool-calling framework to one application's needs. It would break keel's clean boundary: "you give me a prompt, I return a tool call."

## Why Assistant is correct

All prompt assembly already lives in the assistant layer:

| Injection point | File:Line | What it does |
|---|---|---|
| `_build_system_prompt()` | `assistant/runtime.py:393` | Combines user config + keel's tool catalog. Natural place for static grounding (model name). |
| `_render_request()` | `assistant/runtime.py:399` | Flattens LLMRequest into user prompt. Already injects `conversation_id` and `actor_id`. Natural place for dynamic grounding (current timestamp). |
| `extract_facts()` | `assistant/memory/extraction.py:43` | Builds its own system+user prompt for fact extraction. Needs grounding to produce time-aware facts. |
| `extract_procedure()` | `assistant/memory/extraction.py:133` | Same pattern for procedure extraction. |

The plan in `1781391002912-playful-canyon.md` is correct on the architecture. The proposed `build_grounding_context()` helper in `assistant/grounding.py` is a clean, minimal approach.

### 2. `assistant/runtime.py` — `_render_request()` (line 399)

Inject grounding block after `conversation_id`/`actor_id` header (line 404), before memory system messages (line 406). This ensures grounding is always fresh per-request.

```python
lines: list[str] = [
    f"conversation_id: {request.conversation_id}",
    f"actor_id: {request.actor_id}",
]
# ADD:
lines.append("")
lines.append(build_grounding_context(
    model=self.settings.llm.model,
    actor_id=request.actor_id,
))
```

### 3. `assistant/memory/extraction.py`

Add `grounding: str = ""` parameter to `extract_facts()` and `extract_procedure()`. Prepend to system prompt when provided.

### 4. `assistant/memory/service.py`

Generate grounding in `extract_and_store_facts()` and `extract_and_store_procedure()` via `build_grounding_context(actor_id=user_id)` and pass to extraction functions.

### 5. `assistant/runtime.py` — `_build_system_prompt()` (line 393)

Optionally inject static grounding (model name) into the system prompt so the LLM knows its identity even in keel's tool-calling system prompt.
